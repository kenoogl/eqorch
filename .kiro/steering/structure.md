# プロジェクト構造

## 組織方針

EqOrch は、実装より先に仕様を固定する構造を採る。したがって、ディレクトリはコードの種類よりも、意思決定の段階と責務の違いが分かることを優先する。

## ディレクトリパターン

### ローカルメモ
**場所**: `/dev_doc/`  
**目的**: 個人用の検討メモ、下書き、途中整理を置く。Git では追跡しない。  
**例**: 仮説メモ、比較表、雑多な論点整理

### プロジェクト全体のステアリング
**場所**: `/.kiro/steering/`  
**目的**: プロダクト方針、技術判断、構造ルールなど、横断的な基準を置く。  
**例**: `product.md`, `tech.md`, `structure.md`

### Kiro 仕様資産
**場所**: `/.kiro/specs/`  
**目的**: Kiro ワークフローに従う個別仕様を置く。要求、設計、タスク、検証の直接入力として扱う。  
**例**: `eqorch/spec.json`, `eqorch/requirements.md`

## 命名規約

- **ファイル**: Markdown は基本的に `kebab-case.md`
- **仕様ディレクトリ**: `.kiro/specs/eqorch` のように feature 名ベースの `kebab-case`
- **ローカル文書**: 日本語名も許容するが、追跡対象ファイルは原則として英数字ベースを優先する

## 依存と配置の原則

- `dev_doc/` の内容を前提に実装判断しない。必要な内容は追跡対象文書へ昇格させる
- `.kiro/specs/` の仕様は実装作業の直接入力として扱う
- `.kiro/steering/` は全仕様に共通する恒久ルールとして扱う
- 実装コード追加後も、構造説明はファイル一覧ではなく配置ルールで維持する
