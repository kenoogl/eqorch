# 技術スタック

## アーキテクチャ

現時点では、方程式探索の処理系を一体で決め打ちせず、仕様主導で最小構成から決める。先に入出力境界、探索ループ、評価責務を分けて定義し、その後に言語や実行基盤を確定する。

## コア技術

- **言語**: 未決定
- **フレームワーク**: 未決定
- **ランタイム**: 未決定

未決定のまま実装を始めない。技術選定は、対象 MVP の要求を仕様で固定した後に行う。

## 主要ライブラリ

現時点で固定ライブラリはない。ライブラリ選定が将来の構造やテスト方針に影響する場合は、採用前に仕様または設計文書へ理由を書く。

## 開発標準

### 仕様駆動

- 実装前に、要求、設計、タスクをレビュー可能な文書に落とす
- 変更はまず仕様へ反映し、その後にコードへ反映する
- ローカルメモは `dev_doc/`、追跡対象の正式文書は `.kiro/steering/` と `.kiro/specs/` に置く

### 文書言語

- 追跡対象の Markdown 文書は原則として日本語で記述する
- 英語の参照情報を使う場合でも、プロジェクト内の判断と要約は日本語で残す

### Git 運用

- ユーザーが `git-cp` と指示した場合は、関連変更をコミットしてリモートへプッシュする依頼として扱う
- コミット前には、対象差分が今回の作業範囲と一致していることを確認する

### コード品質

- 小さな MVP を優先し、不要な抽象化を避ける
- 境界が曖昧な処理は、まず文書で責務分割を明確化する
- 振る舞い変更には検証手順を残す

### テスト

- テスト方針は技術選定後に決める
- ただし各機能仕様には、少なくとも受け入れ条件と検証方法を記載する

## 開発環境

### 必要ツール

- Git
- Codex / Kiro スタイルの仕様駆動ワークフローを扱える環境

### 共通コマンド

```bash
# 状態確認: git status --short
# 仕様確認: find .kiro/specs -maxdepth 3 -type f | sort
# ステアリング確認: find .kiro/steering -maxdepth 1 -type f | sort
```

## 重要な技術判断

- 初期段階では技術スタックの固定より、仕様の粒度と探索対象の切り分けを優先する
- 実装に入る前に、最初の MVP に必要な入出力、評価指標、反復単位を仕様化する
- 技術判断は再利用性よりも、最初の検証可能性を優先する
